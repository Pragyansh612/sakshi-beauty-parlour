### Step 5 — Auth
- `/login` — tab-switching split-screen page (`app/login/page.tsx` + `components/auth/AuthPanel.tsx`). Login/Register/ForgotPassword as separate client form components (`components/auth/LoginForm.tsx`, `RegisterForm.tsx`, `ForgotPasswordForm.tsx`).
- Per BLOCKER-001: login uses email+password (`signInWithPassword`); register collects full name + phone + email + password, calls `signUp` with `options.data = { full_name, phone }` which the `handle_new_user()` trigger (schema.sql) reads to populate `profiles`.
- Register: if Supabase returns a session immediately (email confirmation disabled), redirect to `/dashboard`; otherwise show "check your email" and switch to login tab.
- Forgot password uses `resetPasswordForEmail`.
- Middleware (`lib/supabase/middleware.ts`) already protects `/dashboard` and `/admin` and redirects logged-in users away from `/login` — this existed from Step 1/2 scaffold, unchanged.

### Step 6 — Booking flows
- `/book` — full wizard matching `design/Booking.dc.html`: mode toggle (Quick Appointment / Bridal & Event Booking), 4-step stepper, review screen, success state. `components/booking/BookingWizard.tsx` (client) holds all wizard state; `app/book/page.tsx` (server) fetches real data.
- Real DB wiring (per Assumption #3 in IMPLEMENTATION_PLAN.md): services fetched from `services` table filtered by `is_appointment_eligible` / `is_booking_eligible`; time slots fetched from `time_slots` for the next 14 days, cross-referenced against non-cancelled `appointments` to grey out already-booked slots (no DB trigger auto-blocks a slot on booking — filtering happens in the page query).
- `actions/appointments.ts`, `actions/bookings.ts` — server actions that insert into `appointments`/`bookings` under the signed-in user; `reference` is auto-generated by existing DB triggers, not set client-side.
- Auth gating: at the Review step, unauthenticated users see a "sign in or register" prompt instead of a confirm button (simplest compliant option from the two the plan allowed — no session-storage/redirect-back was implemented).
- `scripts/seed-slots.ts` (new, mirrors `scripts/seed-images.ts`) — generates open `time_slots` for the next N days (default 60), 11:00 AM–8:30 PM every 30 min. Needed because `schema.sql` seeds services but not slots (slots are date-based and can't be hardcoded); run this once a real Supabase project is connected, or the date/slot pickers will show no availability.
