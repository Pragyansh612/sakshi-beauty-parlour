### Step 6 — Booking flows
- `/book` — full wizard matching `design/Booking.dc.html`: mode toggle (Quick Appointment / Bridal & Event Booking), 4-step stepper, review screen, success state. `components/booking/BookingWizard.tsx` (client) holds all wizard state; `app/book/page.tsx` (server) fetches real data.
- Real DB wiring (per Assumption #3 in IMPLEMENTATION_PLAN.md): services fetched from `services` table filtered by `is_appointment_eligible` / `is_booking_eligible`; time slots fetched from `time_slots` for the next 14 days, cross-referenced against non-cancelled `appointments` to grey out already-booked slots (no DB trigger auto-blocks a slot on booking — filtering happens in the page query).
- `actions/appointments.ts`, `actions/bookings.ts` — server actions that insert into `appointments`/`bookings` under the signed-in user; `reference` is auto-generated by existing DB triggers, not set client-side.
- Auth gating: at the Review step, unauthenticated users see a "sign in or register" prompt instead of a confirm button (simplest compliant option from the two the plan allowed — no session-storage/redirect-back was implemented).
- `scripts/seed-slots.ts` (new, mirrors `scripts/seed-images.ts`) — generates open `time_slots` for the next N days (default 60), 11:00 AM–8:30 PM every 30 min. Needed because `schema.sql` seeds services but not slots (slots are date-based and can't be hardcoded); run this once a real Supabase project is connected, or the date/slot pickers will show no availability.

### Step 8 — Admin panel
- `app/admin/layout.tsx` — server-side auth+role guard (defensive; `proxy.ts` middleware already redirects non-admins) rendering `components/admin/AdminSidebar.tsx` (desktop: sticky column; mobile: hamburger-triggered full-screen drawer, per `IMPLEMENTATION_PLAN.md` §7 "Admin panel is not mobile-first but must not break completely"). `app/admin/page.tsx` redirects to `/admin/dashboard`.
- `components/shared/RecordDrawer.tsx` — extended to actually render `type: 'select'` fields (the interface already declared this option but the previous implementation only ever rendered a text/date `Input`); needed for Status/Category dropdowns across every admin manager so invalid free-text can't be typed into constrained columns.
- `/admin/dashboard` — stats grid (today's appointments, upcoming bookings incl. bridal-this-month, revenue this month, pending approvals), today's schedule table, and a recent-activity feed derived from `created_at`/`cancelled_at` across appointments + bookings (no separate activity-log table exists, so this is computed at read time, capped to the 5 most recent events).
- `/admin/appointments` — `AppointmentsManager` (client): search (client/ref/service) + All/Today/Upcoming segment filter, table, `RecordDrawer` (Artist + Status editable; Client/Service/Date&Time/Reference read-only since they're relational), Cancel routed through `DeleteConfirmModal` → `cancelAppointmentAdmin` (soft cancel: `status='cancelled'`, not a row delete — appointments keep history).
- `/admin/bookings` — `BookingsManager` (client): card list, search, `RecordDrawer` (Artist/Agreed price/Status/Admin notes editable) with the CTA label switching Approve → Manage → Update by current status per the design, Delete routed through `DeleteConfirmModal` → hard `deleteBookingAdmin` (bookings really are row-deleted here, matching `IMPLEMENTATION_PLAN.md`'s explicit "Delete" action, unlike appointments' soft-cancel). Did **not** implement admin-side "create a new booking for a customer" (the design's "+ New booking" button) — it needs a customer picker not specified anywhere in Phase 0 docs and isn't in the P0 list; omitted rather than half-built. Noting `agreed_price` is stored in **paise** per `DATABASE_DESIGN.md` (like `services.price_from`) — the admin form shows/accepts rupees and converts both ways.
- `/admin/services` — `ServicesManager` (client): table, search, `RecordDrawer` doubles as the "Add service" form (blank draft + `isNew` flag → calls `createService` instead of `updateService`), category/status as selects, Delete through `DeleteConfirmModal` (relies on `services.*_id ON DELETE RESTRICT` FK — deleting a service still linked to appointments/bookings surfaces a friendly error instead of a raw Postgres error).
- `/admin/gallery` — `GalleryManager` (client): Our Work/Achievements tab toggle, dashed drop-zone card (drag-and-drop or click-to-browse) uploads directly to the `gallery` Supabase Storage bucket client-side via `lib/supabase/client.ts`, then calls `actions/admin/gallery.ts#createGalleryImage` to insert the DB row (per `DEPLOYMENT_PLAN.md`'s "direct browser-to-Storage upload" pattern) — title is auto-derived from the filename, category/tag default per section, refine via the Edit drawer afterward. `lib/supabase/storage.ts#getGalleryPublicUrl` builds the public CDN URL for card thumbnails. Delete removes both the DB row and the Storage object.
- `/admin/slots` — `SlotsManager` (client) + server page computes a Monday-start week (`?week=` offset search param) and cross-references `time_slots` against non-cancelled `appointments` to classify each cell as Open/Full(booked)/Blocked/none(not yet generated). Clicking an Open or Blocked cell toggles it directly (no confirm modal — it's a reversible, non-destructive status flip, unlike every delete/cancel action elsewhere). "Full" and ungenerated cells aren't clickable. "+ Add slot" opens a small date-picker modal that generates a full day (11:00 AM–8:30 PM, 30-min increments) via `generateDaySlots`, mirroring `scripts/seed-slots.ts`'s logic so future weeks aren't stuck empty.
- Every delete/cancel action across Appointments, Bookings, Services, and Gallery goes through the shared `DeleteConfirmModal` — no separate confirm UI was built anywhere in the admin panel.
- All admin server actions (`actions/admin/*.ts`) re-check `role='admin'` server-side via `requireAdmin()` before mutating — not just relying on the `proxy.ts` middleware — since Server Actions can in principle be invoked directly.
